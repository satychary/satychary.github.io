React

React (to make websites 'reactive') is a framework from Facebook. While it has a different syntax compared to Angular etc., the core ideas remain the same.

React example

Here is a React example: https://codesandbox.io/s/react-new - as you can see, we create our app as a function (called App(){}), which RETURNS HTML!

React - the big idea

MVC (Model-View-Controller) is a classied, tried-and-true architectural pattern, where the 'model' (data/state...) is kept SEPARATE from its 'view' (display/presentation/...) - mediated by a 'controller' [to keep the M and V in sync].

MVC is wonderful, but, NOT necessary in HTML! What if an HTML element (which is [the] 'view') is BUNDLABLE along with its model (eg. JSON data), in the form of a 'template', which makes this "reactive" (when the model updates, so does the view, and vice versa)? So, templates manage their own state changes!

React views

React 'components' declare how a view updates, given model/state changes - the developer does NOT need to manage this (in jQ, she would!).

In other words, a React component maintains its own model/state, updates its view). In other words, it's M+V+C :)

(React) components

React components can contain other components - an app is such an organization (parent-child) of components.

A parent component communicates its state to children, via (read-only) 'props'.

Children communicate with a parent, using the parent's callbacks.

Typically, children components do NOT maintain their own state - they are stateless, render-only - that update, when the parent calls them with (read-only) props.

Virtual DOM

React components do not directly update HTML elements when models change - that would lead to many, constant, updates, making the UI be sluggish. Instead, components update an off-screen, 'virtual' DOM - whose changes efficiently get transferred to the actual DOM.

Here is a barebones example of virtual DOM usage: https://bytes.usc.edu/~saty/tools/xem/run.html?x=ReactEx

'Native' frameworks

If we made an Angular/React/Vue/... webpage app and viewed it using our phone's browser, that would need to use the browser's JavaScript layer, so the app would load and run slower than 'natively' running it.

A native framework helps BYPASS JS entirely, and instead, run directly/natively on the phone's OS, ie. Android or iOS.
